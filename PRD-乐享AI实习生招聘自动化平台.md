# 需求背景与目标

## 需求背景

乐享AI团队长期滚动招聘实习生，岗位要求候选人一般可实习 3 个月以上，最好具备 AI 工具使用、Agent / RAG / MCP / Workflow 等相关经验。目前流程包括发布 JD、邮箱收简历、人工下载简历、人工筛选、电话确认意向、Outlook 邮件预约面试、整理面试表现、电话联系候选人、Offer 沟通和入职材料收集。

当前已经面试 50 人以上、入职 15 人以上，说明招聘需求稳定且流程高频复用。但候选人信息、简历判断、电话记录、面试反馈和入职材料状态较多依赖人工介入，导致以下问题：

- 简历入口分散，邮箱附件、社交媒体线索、招聘平台线索难以形成统一候选人池。
- 筛选标准没有结构化沉淀，重复判断 AI 经验、实习时长、到岗时间和岗位匹配度。
- 电话确认成本高，求职意向、可实习时长、到岗时间、北京线下等基础信息可以前置自动确认。
- 面试反馈难以复用，历史面试与入职结果没有反哺后续候选人排序。
- 入职材料收集依赖人工 checklist，缺失项、格式要求、候选人跟进都容易反复沟通。

本需求目标是建设一个轻量招聘自动化工作台，将候选人从投递到入职的核心信息结构化，并引入阿里云百炼大模型能力完成简历解析、岗位匹配评分、候选人摘要、邮件草稿和面试反馈整理。

## 量化目标

| 指标名 | 目标值 |
|--------|--------|
| 简历初筛耗时 | 单份简历人工阅读时间减少 70%，上线 30 天内达成 |
| 电话前置过滤 | 进入电话沟通前，至少 80% 候选人完成意向确认表 |
| 候选人信息完整率 | 进入面试阶段候选人的关键字段完整率达到 95% |
| 面试预约效率 | 从筛选通过到发出面试邀请的平均耗时小于 1 个工作日 |
| 入职材料返工率 | 因材料缺失或格式错误产生的二次沟通减少 50% |

# 模块A：候选人池

## 原子能力定义

候选人池是招聘流程的主数据模块，负责承载候选人基础信息、来源、简历、筛选结果、沟通记录、面试反馈、Offer 状态和入职材料状态。该模块不负责实际解析简历和发送邮件，但为解析、触达、预约和入职流转提供统一数据底座。

建议一期可使用飞书多维表格或轻量 Web 工作台承载数据。若已有内部工具限制，优先采用飞书多维表格作为数据层，后续再升级为独立 Web 应用。

## 核心字段

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| 候选人ID | 文本 | 必填 | 自动生成，避免同名候选人冲突 |
| 姓名 | 文本 | 必填 | 从邮件标题、简历或表单提取 |
| 学校 | 文本 | 选填 | 简历解析或候选人补充 |
| 学历 | 单选 | 选填 | 本科 / 硕士 / 博士 / 其他 |
| 专业 | 文本 | 选填 | 计算机、工业设计等可重点标记 |
| 邮箱 | 文本 | 必填 | 投递邮箱或表单邮箱 |
| 手机号 | 文本 | 选填 | 电话沟通使用 |
| 来源渠道 | 单选 | 必填 | 邮箱 / 小红书 / 招聘平台 / 内推 / 其他 |
| 简历文件 | 附件 | 必填 | 原始简历 |
| AI工具经验 | 多选 | 选填 | ChatGPT / Claude / Gemini / DeepSeek / 其他 |
| LLM知识点 | 多选 | 选填 | RAG / SFT / Agent / MCP / Context Engineering / Workflow |
| 项目经验摘要 | 长文本 | 选填 | 乐享从简历中提取 |
| 可实习时长 | 数字 | 必填 | 单位：月 |
| 最早到岗时间 | 日期 | 必填 | 候选人确认 |
| 每周可出勤 | 数字 | 选填 | 单位：天 |
| 是否接受北京线下 | 单选 | 必填 | 是 / 否 / 待确认 |
| 简历评分 | 数字 | 选填 | 0-100 |
| 推荐等级 | 单选 | 选填 | 强推 / 可面 / 备选 / 不建议 |
| 当前状态 | 单选 | 必填 | 见状态机 |
| 面试时间 | 日期时间 | 选填 | 已约面候选人填写 |
| 面试反馈 | 长文本 | 选填 | 面试官录入 |
| 综合排序 | 数字 | 选填 | 结合简历与面试得分 |
| Offer状态 | 单选 | 选填 | 未发 / 已发 / 已接受 / 已拒绝 |
| 入职材料状态 | 单选 | 选填 | 未开始 / 待补充 / 已齐全 / 已提交 |

## 核心状态机

| 状态 | 描述与处理逻辑 |
|------|----------------|
| 新投递 | 候选人简历已进入候选人池，等待解析 |
| 待筛选 | 简历已上传，等待乐享解析与评分 |
| 筛选通过 | 简历评分满足阈值，准备发送意向确认 |
| 待意向确认 | 已发送确认表，等待候选人补充到岗、时长、线下等信息 |
| 待电话沟通 | 基础条件满足，但仍需人工电话确认 |
| 待预约面试 | 电话或表单确认通过，等待确定面试时间 |
| 已预约面试 | 已发送面试邀请，等待面试 |
| 已面试 | 面试已完成，等待反馈和排序 |
| 待Offer | 候选人排名靠前，准备 Offer 沟通 |
| 已Offer | 已完成 Offer 沟通，等待候选人接受或入职材料 |
| 待入职材料 | 候选人接受 Offer，等待提交材料 |
| 已入职 | 已完成入职流程 |
| 备选 | 候选人暂不优先，但保留后续联系 |
| 淘汰 | 不满足岗位要求或流程终止 |
| 放弃 | 候选人主动放弃 |

## 交互入口与触发逻辑

### 候选人列表

招聘负责人进入候选人列表后，可以按状态、来源、推荐等级、可实习时长、到岗时间筛选候选人。点击候选人可查看简历解析摘要、原始简历、沟通记录、面试反馈和入职材料状态。

### 候选人详情

详情页展示候选人完整履历、乐享评分解释、风险点、推荐下一步动作。负责人可手动调整状态、补充电话记录、生成邮件草稿或发起入职材料收集。

## 场景细分

### 成功场景

- **触发意图：** 招聘负责人查看“待筛选”候选人。
- **关键信息提取：**
  - `candidate_id`：（必选）候选人 ID。
  - `resume_file`：（必选）候选人简历文件。
  - `job_profile`：（必选）当前岗位 JD 与评分标准。
- **预期行为：** 乐享展示候选人摘要、匹配评分、推荐等级和下一步动作。

> ⚠️勿写死，由大模型自主生成，以下只是举个例子  
> 该候选人具备 Claude 与 DeepSeek 高频使用经验，参与过校园知识库项目，对 RAG 有基础理解；可实习 4 个月，最早 6 月 10 日到岗。建议进入意向确认，重点追问 Agent / Workflow 的实际配置经验。

### 失败/边界场景

- **条件：** 简历缺少联系方式或无法解析。
- **乐享回答：** 乐享提示缺失字段，并建议从邮件正文、附件文件名或人工补充入口完善信息。

# 模块B：简历解析与岗位匹配评分

## 原子能力定义

简历解析与岗位匹配评分负责将候选人简历转为结构化数据，并根据乐享AI团队岗位要求输出评分、推荐等级和风险备注。该模块依赖大模型能力，建议接入阿里云百炼兼容 OpenAI 或 Anthropic 协议的接口。

该模块只提供辅助判断，不直接做最终淘汰决策。所有自动评分均需可解释，并保留人工调整入口。

## 核心状态机

| 状态 | 描述与处理逻辑 |
|------|----------------|
| 待解析 | 已收到简历，尚未调用解析能力 |
| 解析中 | 乐享正在抽取简历文本并调用大模型 |
| 解析成功 | 已产出结构化字段、评分、推荐等级 |
| 解析失败 | 文件损坏、文本为空、格式异常或接口失败 |
| 待人工复核 | 评分低置信或关键信息缺失，需要人工确认 |

## 评分规则

| 评分项 | 权重 | 说明 |
|--------|------|------|
| AI工具深度 | 25% | 是否重度使用 ChatGPT / Claude / Gemini / DeepSeek，是否能说明实际使用场景 |
| 产品与项目经历 | 20% | 是否有需求分析、竞品调研、原型设计、产品落地经验 |
| LLM基础认知 | 20% | 是否了解 RAG、SFT、Context Engineering、Agent、MCP、Skills 等概念 |
| 数据处理与评测经验 | 15% | 是否有数据清洗、质量控制、大模型评测、知识库建设经验 |
| 实习稳定性 | 15% | 是否能线下到岗、可实习 3 个月以上、到岗时间匹配 |
| 表达与简历质量 | 5% | 简历结构、表达清晰度、成果描述可信度 |

推荐等级：

| 分数 | 等级 | 处理建议 |
|------|------|----------|
| 85-100 | 强推 | 优先意向确认并尽快约面 |
| 70-84 | 可面 | 进入意向确认，根据时长和到岗时间排序 |
| 55-69 | 备选 | 保留候选人，岗位紧急时补充联系 |
| 0-54 | 不建议 | 原则上不进入面试，除非人工标记例外 |

## 交互入口与触发逻辑

### 上传或同步简历后自动解析

候选人记录进入“待筛选”状态后，乐享自动读取附件，提取文本并调用百炼服务，生成候选人结构化摘要。

## 场景细分

### 成功场景

- **触发意图：** 新简历进入候选人池。
- **关键信息提取：**
  - `resume_text`：（必选）简历正文。
  - `job_description`：（必选）岗位 JD。
  - `scoring_rubric`：（必选）评分规则。
- **预期行为：** 输出 JSON 结构化结果并写回候选人池。

输出字段：

```json
{
  "candidate_name": "",
  "school": "",
  "major": "",
  "degree": "",
  "ai_experience_summary": "",
  "tool_usage": [],
  "llm_knowledge": [],
  "product_experience": "",
  "data_work_experience": "",
  "available_months": null,
  "arrival_date": "",
  "risk_notes": "",
  "score": 0,
  "recommendation": "强推/可面/备选/不建议",
  "interview_focus": []
}
```

### 失败/边界场景

- **条件：** 简历无法解析出姓名、联系方式或实习时长。
- **乐享回答：** 乐享将候选人状态设为“待人工复核”，并展示缺失字段，提醒负责人通过邮件正文或电话补充。

## 工具调用

##### 简历解析工具：

输入：
- `resume_file`（必选）：PDF / DOCX / 图片简历文件。
- `job_description`（必选）：岗位 JD 文本。
- `scoring_rubric`（必选）：评分规则。

输出：
- `parsed_profile`：候选人结构化资料。
- `score_result`：评分、推荐等级、评分解释。
- `risk_notes`：风险点与建议追问问题。

##### 阿里云百炼大模型服务：

输入：
- `base_url`（必选）：通过服务端环境变量配置。
- `api_key`（必选）：通过服务端密钥管理配置，不进入前端、不写入文档正文和日志。
- `model`（必选）：按百炼可用模型配置。
- `messages`（必选）：岗位 JD、评分规则、简历文本和输出格式约束。

输出：
- `structured_json`：符合候选人字段定义的 JSON。
- `confidence`：解析置信度。
- `raw_response_id`：接口响应 ID，用于排查问题。

接入配置：

| 协议 | Base URL | 用途 |
|------|----------|------|
| OpenAI 兼容协议 | `https://coding.dashscope.aliyuncs.com/v1` | 简历解析、评分、邮件草稿、反馈整理 |
| Anthropic 兼容协议 | `https://coding.dashscope.aliyuncs.com/apps/anthropic` | 如现有工具链使用 Anthropic SDK，可走此协议 |

安全要求：

- API Key 只允许保存在服务端环境变量或云端密钥管理中。
- 日志中禁止打印 API Key、候选人身份证号、银行卡号等敏感信息。
- 简历原文和入职材料属于个人信息，必须设置访问权限和留存周期。

# 模块C：意向确认与电话沟通前置

## 原子能力定义

意向确认模块负责在电话沟通前自动收集候选人的求职意向、实习时长、到岗时间、每周出勤、是否接受北京线下和可面试时间，降低无效电话比例。

## 核心状态机

| 状态 | 描述与处理逻辑 |
|------|----------------|
| 待发送 | 候选人筛选通过，等待发送意向确认 |
| 已发送 | 已通过邮件或表单链接发送确认请求 |
| 已填写 | 候选人已提交确认信息 |
| 条件满足 | 实习时长、到岗时间、线下要求满足岗位要求 |
| 条件不满足 | 不满足硬性要求，进入备选或淘汰 |
| 超时未回复 | 发送后 2 个工作日未回复，提醒一次；仍未回复进入低优先级 |

## 交互入口与触发逻辑

### 自动发送意向确认邮件

当候选人推荐等级为“强推”或“可面”时，负责人可批量选择候选人并发送意向确认邮件。邮件中包含表单链接，候选人填写后自动回写候选人池。

## 场景细分

### 成功场景

- **触发意图：** 批量处理筛选通过候选人。
- **关键信息提取：**
  - `candidate_email`：（必选）候选人邮箱。
  - `candidate_name`：（可选）候选人姓名。
  - `confirmation_form_url`：（必选）意向确认表链接。
- **预期行为：** 乐享生成并发送意向确认邮件，回收表单后更新状态。

意向确认表字段：

| 字段 | 类型 | 是否必填 |
|------|------|----------|
| 姓名 | 文本 | 必填 |
| 手机号 | 文本 | 必填 |
| 是否仍有求职意向 | 单选 | 必填 |
| 最早到岗时间 | 日期 | 必填 |
| 可连续实习几个月 | 数字 | 必填 |
| 每周可到岗几天 | 数字 | 必填 |
| 是否接受北京联想总部东区线下实习 | 单选 | 必填 |
| 近期可面试时间段 | 多行文本 | 必填 |
| 补充说明 | 多行文本 | 选填 |

> ⚠️勿写死，由大模型自主生成，以下只是举个例子  
> Hi 同学，感谢投递联想乐享AI团队实习岗位。为了更快安排后续流程，辛苦先补充一下到岗时间、可实习时长和近期可面试时间。我们会根据你的回复尽快安排下一步沟通。

### 失败/边界场景

- **条件：** 候选人填写可实习时长小于 3 个月。
- **乐享回答：** 乐享将候选人标记为“备选”，并提示负责人如岗位短期缺口紧急，可人工调整进入电话沟通。

## 工具调用

##### 意向确认邮件生成工具：

输入：
- `candidate_name`（可选）：候选人姓名。
- `position_name`（必选）：岗位名称。
- `form_url`（必选）：确认表链接。

输出：
- `email_subject`：邮件标题。
- `email_body`：邮件正文。

# 模块D：面试预约与邮件模板

## 原子能力定义

面试预约模块负责根据候选人确认的可面试时间，辅助负责人生成 Outlook 面试邀请邮件，并记录面试时间、面试方式和面试状态。一期不强依赖 Outlook Calendar API，可先生成邮件草稿；二期再接入日历空闲时间查询与自动发邀。

## 核心状态机

| 状态 | 描述与处理逻辑 |
|------|----------------|
| 待预约 | 候选人已确认意向，等待安排面试 |
| 候选人时间已收集 | 已获取候选人可面时间段 |
| 邮件草稿已生成 | 乐享生成面试邀请邮件 |
| 已发送邀请 | 负责人确认后发送 Outlook 邮件 |
| 候选人已确认 | 候选人确认参加面试 |
| 已面试 | 面试已完成，进入反馈整理 |
| 改期 | 候选人或面试官需要调整时间 |
| 未参加 | 候选人缺席或临时放弃 |

## 交互入口与触发逻辑

### 面试邀请生成

负责人在候选人详情页点击“生成面试邀请”，乐享根据候选人姓名、面试时间、岗位名称和面试链接生成邮件正文。

## 场景细分

### 成功场景

- **触发意图：** 为候选人发送面试邀请。
- **关键信息提取：**
  - `candidate_name`：（必选）候选人姓名。
  - `interview_time`：（必选）面试时间。
  - `interview_link`：（必选）面试链接或会议室信息。
  - `contact_phone`：（必选）联系人电话。
- **预期行为：** 乐享生成正式、简洁的面试邀请邮件。

面试邀请模板：

```text
Hi {候选人姓名}，

感谢投递联想，以下是 AI 产品实习生的面试邀请，请按沟通时间进入面试间。

面试时间：{面试日期时间}
面试方式：{面试链接或地点}

Best wishes
{联系人姓名}
{联系人电话}
```

### 失败/边界场景

- **条件：** 候选人可面时间与面试官时间冲突。
- **乐享回答：** 乐享提示冲突，并建议向候选人发送补充可面时间请求。

## 工具调用

##### 面试邀请邮件工具：

输入：
- `candidate_name`（必选）：候选人姓名。
- `interview_time`（必选）：面试时间。
- `interview_location_or_link`（必选）：面试地点或链接。
- `contact_phone`（必选）：联系人电话。

输出：
- `email_subject`：面试邀请标题。
- `email_body`：面试邀请正文。

# 模块E：面试反馈与候选人排序

## 原子能力定义

面试反馈模块负责录入面试表现、整理候选人优势与风险、结合简历评分输出综合排序。该模块帮助招聘负责人快速决定 Offer 顺序，但不替代面试官决策。

## 核心状态机

| 状态 | 描述与处理逻辑 |
|------|----------------|
| 待反馈 | 候选人已面试，等待面试官录入反馈 |
| 反馈已录入 | 已填写面试表现与建议 |
| 待排序 | 等待和同批候选人比较 |
| 排序完成 | 已形成推荐顺序 |
| 待Offer | 排序靠前，准备电话沟通 Offer |
| 备选观察 | 表现尚可但非首选 |
| 淘汰归档 | 不进入后续流程 |

## 面试反馈字段

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| 面试官 | 文本 | 必填 | 记录反馈人 |
| 表达与结构化 | 评分 | 必填 | 1-5 分 |
| AI工具使用深度 | 评分 | 必填 | 1-5 分 |
| LLM认知 | 评分 | 必填 | 1-5 分 |
| 产品sense | 评分 | 必填 | 1-5 分 |
| 项目推进能力 | 评分 | 必填 | 1-5 分 |
| 稳定性 | 评分 | 必填 | 1-5 分 |
| 亮点 | 长文本 | 必填 | 候选人优势 |
| 风险 | 长文本 | 必填 | 需要注意的问题 |
| 面试结论 | 单选 | 必填 | 强推 / 可发Offer / 备选 / 不通过 |

## 交互入口与触发逻辑

### 面试后反馈整理

面试官可直接填写结构化表单，也可粘贴自然语言面试记录，由乐享整理成统一反馈格式。

## 场景细分

### 成功场景

- **触发意图：** 面试官粘贴面试记录。
- **关键信息提取：**
  - `interview_notes`：（必选）面试原始记录。
  - `resume_score`：（必选）简历评分。
  - `rubric`：（必选）面试评分规则。
- **预期行为：** 乐享整理候选人亮点、风险、综合评分和推荐结论。

> ⚠️勿写死，由大模型自主生成，以下只是举个例子  
> 候选人表达清晰，对 AI 工具使用频率较高，能说明 DeepSeek 辅助资料整理的具体流程；但对 RAG 与 Agent 的理解偏概念化，缺少实际配置经验。建议作为可发 Offer 候选，入职后优先安排知识库和评测任务。

### 失败/边界场景

- **条件：** 面试记录过短，无法支撑排序。
- **乐享回答：** 乐享提示反馈信息不足，并引导补充“项目案例、AI 工具使用、实习稳定性、面试结论”四项。

## 工具调用

##### 面试反馈整理工具：

输入：
- `interview_notes`（必选）：面试官原始记录。
- `candidate_profile`（必选）：候选人结构化资料。
- `feedback_rubric`（必选）：反馈评分规则。

输出：
- `summary`：面试摘要。
- `strengths`：亮点。
- `risks`：风险。
- `final_recommendation`：最终建议。
- `next_action`：下一步动作。

## 场景细分

# MVP 范围

## 一期必须做（P0）

- 候选人池字段与状态流转。
- 简历上传、解析、摘要和评分。
- 意向确认表与候选人状态回写。
- 面试邀请邮件模板生成。
- 面试反馈结构化录入。
- Offer 信息与入职材料 checklist。
- 百炼服务端接入，密钥通过环境变量管理。

## 一期暂不做（P1/P2）

- 自动拨打电话或语音外呼。
- 完全自动淘汰候选人并发送拒信。
- 自动读取 Outlook 全量日历并发起会议邀请。
- 基于历史入职表现训练专属模型。
- 多岗位、多团队、多 HR 复杂权限。

# 风险与待确认问题

| 问题 | 影响 | 建议 |
|------|------|------|
| Outlook 邮箱和日历 API 权限未确认 | 影响自动下载简历、自动发邀 | 一期先做邮件草稿和人工确认，二期接 Microsoft Graph |

| 招聘渠道入口多 | 影响线索自动汇总 | 一期优先邮箱入口，社媒和招聘平台用人工导入或表单承接 |
