# Release: 通用招聘项目工作台并行预览环境

发布日期：2026-05-29

## 发布类型

并行测试环境 / Preview，不替换现有线上招聘工作台。

该页面是静态 mock 预览，用于体验信息架构和界面逻辑。页面按钮不会写入真实候选人数据，也不会触发飞书、Outlook、Graph、Teams 或邮件发送。

## 预览入口

- 现有正式入口保持不变：`https://new.leaibot.cn/recruiting/`
- 新增预览入口：`https://new.leaibot.cn/recruiting/previews/generic-recruiting-platform-preview.html`

## 本次范围

- 新增一个静态 HTML 预览页，用于评审“通用招聘项目工作台”的产品逻辑。
- 首屏聚焦当前招聘项目、项目候选人池、候选人当前旅程和下一步动作。
- 下方评审区展示项目配置、导入来源配置、候选人总库与项目投递关系的数据模型。
- 修正语音识别误解：页面中不使用“星社来源”，统一表达为“新设置的来源”或“新设置的导入配置”。

## 不在本次范围

- 不修改现有 React 工作台业务逻辑。
- 不新增后端 API、数据库表、权限模型或真实数据迁移。
- 不改变当前乐享AI实习生招聘的线上流程、登录、候选人数据或面试安排能力。
- 不同步 `.env`、`data/`、`uploads/`、`artifacts/`，不读取或暴露真实候选人附件。
- 不把预览页作为最终方案直接上线，后续替换前需要单独做实现方案和数据迁移评估。

## 操作边界

- `新建招聘项目`、`项目设置`、`新增来源`、`进入面试安排` 等按钮只用于表达未来交互入口。
- 当前预览数据均为示意数据，不代表真实候选人、真实招聘状态或真实数据总量。
- 正式生产主链路仍是现有工作台：飞书投递、AI 筛选、人工通过、面邀、Outlook/Teams、面评、Offer。
- 通用项目工作台只验证 IA 与体验，不默认迁移历史数据。

## 评审重点

- 平台是否应该从“单次乐享AI招聘工具”升级为“多团队、多岗位、多项目可复用的招聘工作台”。
- “候选人总库”和“项目候选人池”是否足够清晰：人只存一份，某次岗位申请按项目独立记录。
- “项目配置驱动”是否成立：团队、岗位、JD、来源、流程、模板和权限都由新建项目时配置，不写死在系统标签里。
- 首页是否足够清爽：第一屏只解决当前项目的工作状态，不把底层模型和新建表单全部压在首屏。

## 验证清单

- 本地构建：`npm run build`
- 本地预览页检查：`/previews/generic-recruiting-platform-preview.html`
- 线上预览页检查：预览入口返回 200，页面包含“招聘项目工作台”和“新设置的来源”。
- 线上正式入口检查：`/recruiting/` 仍返回 200，现有工作台不被替换。
- 数据安全检查：确认本次发布只新增静态预览文件和 Release 文档。

## 回滚方式

如果预览页不再需要，删除 `public/previews/generic-recruiting-platform-preview.html` 后重新发布即可；正式入口不受影响。

## 后续替换条件

- 确认信息架构：workspace / requisition / candidate / application 四层模型。
- 确认新建项目、来源配置、字段映射和权限协作的真实交互。
- 评估现有乐享AI数据如何迁移为第一个招聘项目。
- 完成 React 工作台实现、后端数据结构调整、回归测试和正式发布文档。
