# 乐享AI实习生招聘自动化系统 — 设计文档

**日期**：2026-05-07
**作者**：陈百科 + Claude
**状态**：待评审

## 0. 背景

乐享AI团队长期滚动招聘实习生（3个月以上，优先AI经验），累计面试50+、入职15+。当前流程人工成本高，涉及：JD发布 → 邮箱收简历 → 人工筛选 → 电话沟通确认意向/时长/到岗 → Outlook约面试 → 面试评估排序 → 入职材料收集。

**目标**：降低从简历到面试决策阶段的人工介入，面试材料收集阶段暂不纳入本期 MVP。

## 1. 需求与范围

### 1.1 本期覆盖

- 简历自动拉取与 LLM 四维度评分
- 自动发问卷（替代电话沟通）
- 候选人自助预约面试时段
- 面试后简短打分 + LLM 辅助小结 + 排序决策
- Offer/入职指引邮件（简单模板发送）

### 1.2 本期不覆盖

- 入职材料扫描件收集与校验
- 多岗位/多面试官复杂排班
- 候选人画像沉淀与回流
- JD 多渠道自动发布
- 简历反作弊

### 1.3 用户

主要管理员一人（陈百科），日常只在飞书多维表格内操作。

## 2. 架构总览

```
[联想Outlook邮箱]  ──(Graph API)──►  ┌─────────────────────┐
                                       │  Python 调度服务     │
[飞书问卷回填]     ──(Webhook)──►      │  (cron / 云函数)     │
                                       │                      │
[飞书/Outlook日历] ──(读空档)──►        │  - 邮件抓取          │
                                       │  - 简历解析 (PDF/DOC)│
                                       │  - LLM 四维打分      │
                                       │  - 时段匹配          │
                                       │  - 发邮件/建会议     │
                                       └──────┬───────────────┘
                                              │ 读/写
                                              ▼
                         ┌────────────────────────────────────────┐
                         │  飞书多维表格 (Base)                    │
                         │  表1: 候选人主表                        │
                         │  表2: 面试记录                          │
                         │  表3: 面试时段池                        │
                         │  表4: JD/岗位配置                       │
                         └────────────────────────────────────────┘
                                       ▲
                                       │ 浏览 / 自动化触发
                                       ▼
                             [管理员 - 招聘决策]
```

核心原则：**飞书 Base 是唯一真源**，Python 服务承担 I/O 和 LLM 调用，飞书自动化承担简单通知触发。管理员日常只在 Base 内看评分、改状态、填面试打分。

### 2.1 技术选型

- **数据层**：飞书多维表格（Base）
- **编排层**：Python 服务（FastAPI + APScheduler），本地 cron 起步，稳定后迁云函数
- **LLM**：Dashscope（阿里云，数据不出境，兼容 OpenAI 协议）—— API Key 放 `.env`，不入库
- **邮件**：Microsoft Graph API 读取 Outlook 邮箱、发送面试邀请
  - 授权方式：使用 Delegated Permissions（用户代理模式），由管理员账号 OAuth 授权一次，refresh_token 存 `.env`
  - 若联想 AAD 禁用个人 OAuth，需申请 Application Permission 并走企业 IT 审批（计入风险项）
- **问卷**：飞书问卷 + Webhook 回调
- **会议**：默认飞书视频会议（与 Base 生态一致，创建接口稳定）；若管理员明确需要 Outlook 会议可在 V2 切换

## 4. 核心流程

### 4.1 流程A：简历进入 → LLM初筛（全自动，每15分钟）

```
[cron]
  → Graph API 拉取 chenbk1@lenovo.com 未读邮件，过滤标题含「超级智能体实习申请」
  → 下载 PDF/DOC 附件到飞书云空间"简历"文件夹
  → pdfplumber / python-docx 抽文本
  → 正则脱敏（身份证、银行卡、手机中段）
  → 一次 LLM 调用返回 JSON（含结构化字段 + 四维评分 + 评语依据）
  → 写入候选人主表，状态=「待人工初筛」
  → 标记邮件已读
  → 异常 → 飞书机器人告警 + 状态「LLM解析失败」
```

去重规则：邮箱 + 投递日期（同一天同邮箱重复投递只留最新一份）。

硬门槛不通过 → 状态直接设为「已婉拒」+ 评语中注明原因。

### 4.2 流程B：初筛通过 → 发问卷 → 约面试（半自动）

```
管理员在 Base 中把状态改为「已发问卷」
  ↓
飞书 Base 自动化触发 → 调 Python 服务 API
  → 生成候选人专属问卷链接
  → 查询表3空闲时段 + 读飞书日历空档（10个工作日内、排除24h内）
  → 填充面试邀请邮件模板（含问卷链接）
  → 通过 Outlook 发送
  ↓
候选人填写问卷提交 → 飞书问卷 Webhook → Python 服务
  → 事务性锁定时段（写表3，主键唯一冲突防重）
  → 创建飞书视频会议 / Outlook 会议并发邀请
  → 回填「预约面试时间」，状态改「已约面试」
  → 飞书消息通知管理员
```

**问卷字段**：求职意向确认、实际可实习时长、到岗日期、从可用时段中选一个。

**兜底**：48h 未填 → 提醒；72h 未回 → 状态「已流失」。

### 4.3 流程C：面试完成 → 评估汇总 → 决策（半自动）

```
面试官在表2填 4 项打分 + 关键观察（30秒内完成）
  ↓
飞书自动化触发 → Python 服务
  → LLM 读取简历亮点 + 打分 + 观察笔记
  → 生成 3-5 句面试小结，写回"LLM面试小结"
  → 候选人状态改「待决策」
  ↓
管理员在"待决策"视图按综合分排序，手动改「终轮结果」
  ↓
自动化触发 → 发 Offer 邮件 + 入职材料清单 + 入职指引 wiki 链接
  → 状态改「已发Offer」
```

LLM 不做录用决策，只输出"推荐级别"，必须输出隐忧段落避免一味捧人。

## 5. 面试时段调度

### 5.1 时段来源

- **固定兜底**：脚本每周生成未来 2 周的周二/周四 14:00-17:00 每 30 分钟一个 slot，存表3
- **动态日历**：发问卷时实时读管理员飞书/Outlook 日历，找 10:00-12:00 与 14:00-18:00 间的 30 分钟连续空档；不落库，只用于渲染问卷

### 5.2 展示逻辑

问卷加载时组合两类 slot，排除 24 小时内的、已占用的、与日历冲突的，按时间升序取最早 8-10 个。

### 5.3 并发冲突处理

问卷提交时 Python 服务：

1. 查表3该时段是否仍为"空闲"
   - 固定 slot：直接读行
   - 日历动态 slot：INSERT 一条新记录（唯一键=时段），冲突即失败
2. 占用成功 → 建会议、发邀请
3. 占用失败 → 问卷返回"已被抢占，请重选"，表单刷新

### 5.4 临时关闭

管理员可手动把表3的 slot 状态改为"已关闭"，下次生成问卷时即不展示。

## 6. LLM Prompt 设计

两处 Prompt 均存于表4的字段中，便于不改代码调整。

### 6.1 Prompt 1：简历解析 + 四维评分

**输入**：系统 Prompt（JD + 评分维度定义 + JSON Schema）+ 用户 Prompt（脱敏后简历文本）

**评分维度（每维度要求"每分对应一条可验证证据"）**：

- AI经验分（0-10）：AI 工具使用深度、Agent/RAG/知识库经验、大模型评测经验
- 逻辑表达分（0-10）：结构清晰度、STAR、量化结果、语言精炼
- 项目深度分（0-10）：demo 级 vs 有真实用户/数据；与 AI 产品的相关性
- 硬门槛（bool）：学历≥本科 AND 时长≥3月 AND 有手机邮箱

**输出约定**：

- 严格 JSON，含 `<维度>分` + `<维度>分依据`
- 缺字段或解析失败 → Python 端 pydantic 校验失败 → 重试 1 次 → 仍失败置状态「LLM解析失败」并告警
- 信息不足时偏向扣分而非猜测；硬门槛字段缺失 → 标 false + 评语写"待确认"，人工介入

### 6.2 Prompt 2：面试小结

**输入**：JD 原文 + 简历 LLM 评语 + 四维打分 + 面试官观察

**输出结构**（plain text）：

- 一句话综合判断
- 2-3 条亮点（优先面试新获信息）
- 1-2 条隐忧（强制输出）
- 推荐结论（强推/推荐/待定/不推，不是录用决策）

## 7. 部署与运行

### 7.1 目录结构

```
/Users/chenbaike/Documents/联想工作/LeAI-Demo/实习生招聘/app
  ├─ scheduler.py         每 15min 跑流程A
  ├─ webhook_server.py    FastAPI，监听飞书 Webhook (8080)
  ├─ llm_client.py        Dashscope 封装
  ├─ outlook_client.py    Graph API 封装
  ├─ lark_client.py       飞书 Base/日历/消息封装
  ├─ prompts/             Prompt 模板本地缓存（主存在表4）
  ├─ .env                 密钥（gitignore）
  └─ logs/                脱敏日志
```

### 7.2 运行形态

- **MVP 阶段**：本地 Mac + cron + ngrok（或联想内部反向代理）暴露 Webhook 端口
- **稳定后**：迁联想内部云函数 / K8s（非 MVP 范围）

### 7.3 敏感信息处理

简历含身份证、手机、银行卡，属 PII + 金融敏感。

1. Dashscope 境内服务，数据不出境
2. **发给 LLM 前必须正则脱敏**：身份证保留前 6 位、银行卡全遮盖、手机中段遮盖
3. 简历 PDF 存飞书云空间，不落本地
4. API Key 走 `.env` + `python-dotenv`，`.gitignore` 覆盖
5. 日志不含 PII，只记候选人 ID 与状态变化

### 7.4 监控告警

- 心跳：scheduler 每次跑完写时间戳到表4；>30min 无更新 → 飞书机器人告警
- 异常：Python 捕获异常 → 飞书消息（含错误摘要 + 候选人 ID）
- LLM 失败：retry 1 次 → 状态「LLM解析失败」→ 告警

### 7.5 依赖

```
fastapi, uvicorn, apscheduler
pdfplumber, python-docx
msgraph-sdk (或 exchangelib)
lark-oapi
openai  # 调 Dashscope 兼容端点
pydantic, python-dotenv, loguru
```

### 7.6 成本估算

按 qwen-max 量级：简历筛选 ~¥0.05/份，面试小结 ~¥0.03/份。按 50 份简历 + 15 场面试/月 → **~¥3/月**，可忽略。

## 8. MVP 阶段划分

### MVP（1-2周）：自动筛简历 + 手动约面试

- 4 张 Base 表建成
- 流程 A 全打通
- 面试邀请邮件手动触发模板发送
- 面试后手填打分 + LLM 小结
- 飞书消息告警

**验收**：10 份真实简历跑通，LLM 评分抽查准确率 > 80%。

### V2（3-4周）：自动问卷 + 时段锁定

- 飞书问卷 + Webhook
- 表3固定 slot 池
- 并发锁 + 会议创建

**验收**：候选人自助预约，无需人工约。

### V3（5-6周）：日历动态 + Offer流

- 日历空档动态读取
- Offer 邮件 + 入职材料清单 + 指引链接
- Base Dashboard 视图（投递数/转化漏斗）

### V3 后（不承诺时间）

- 多岗位/多面试官
- 候选人画像回流
- 小红书等渠道 JD 自动发布
- 入职材料表单收集

## 9. YAGNI 明确不做

- 通用 HR SaaS
- 多租户 / 权限体系

- 简历反作弊 / 查重

## 10. 风险与备选

| 风险 | 影响 | 应对 |
|---|---|---|
| Graph API 企业权限审批慢 | 流程 A 卡住 | MVP 先支持手动导出 `.eml` 拖入触发；后补 Graph API |
| 飞书 Base 自动化触发频率限制 | 高峰期延迟 | 非实时场景容忍延迟；核心流程靠 Python scheduler 自控 |
| 本地 cron 断电/重启 | 漏拉邮件 | 心跳告警 + 每次启动补拉过去 24h 邮件 |
| LLM 评分偏差 | 误筛 | "待人工初筛"状态保留人工决策权；评语必须含依据 |
| 候选人简历不是 PDF/DOC | 解析失败 | MVP 只支持 PDF/DOC/DOCX；其他类型告警人工处理 |

## 11. 成功指标

- 管理员每个候选人投入时间：当前约 20 分钟 → 目标 < 5 分钟
- 简历到面试邀请平均时长：当前约 2-3 天 → 目标 < 1 天
- 候选人无效投递过滤率 > 95%（硬门槛不过自动走「已婉拒」）
- LLM 评分与管理员人工复核一致率 > 80%
